## Общая архитектура

- **Бэкенд**: использовать текущий `server.js` + `admin-routes` как основу, расширяя REST‑API и схемы БД миграциями в `migrations/*.sql`.
- **Фронтенд админки**: дорабатывать существующий SPA в `admin-subdomain` (React/Vue/vanilla, по текущей реализации) — новые экраны и компоненты поверх существующей структуры, без затрагивания публичного сайта.
- **Данные**: хранить проекты, статьи, отзывы, клиентов и заказы в уже существующих таблицах `projects`, `project_locales`, `articles`, `article_locales`, `clients`, `orders` + новые таблицы для отзывов и расширенной аналитики.

## 1. Дашборд (аналитика)

- **1.1. Хранилище и настройки интеграций**
  - В `settings` добавить ключи для Яндекс Метрики и Вебмастера (ID счётчика, OAuth/токены, при необходимости URL‑ы API).
  - В админ‑панели в настройках сделать блок "Интеграции" с формами для ввода/редактирования этих ключей.
- **1.2. Backend‑обёртка поверх Яндекс API**
  - Реализовать сервис/модуль, который по запросу админки ходит в API Метрики и возвращает агрегированные данные: просмотры, клики, устройства, браузеры, пол, возраст, гео.
  - Добавить кэширование ответов (например, 5–15 минут) во избежание превышения лимитов API.
- **1.3. Эндпоинты дашборда**
  - Добавить в `admin-routes` `/api/admin/analytics/overview`, `/devices`, `/geo`, `/demography` и т.п. с чётко описанным форматом ответа.
- **1.4. UI дашборда**
  - Спроектировать главный экран админки: несколько виджетов (карточки KPI, линейные/столбчатые графики, пайча́рты, таблица ТОП‑городов/стран).
  - Для гео — реализовать вид "карта + таблица" (интеграция с картой через iframe/JS‑библиотеку, без хранения тайлов у себя).
- **1.5. Экспорт отчётов**
  - На каждый виджет добавить кнопки экспорта: **PNG/JPEG** (рендер компонента в картинку), **PDF** (генерация PDF из набора виджетов), **Excel** (выгрузка таблиц в `.xlsx`).
  - Реализовать возможность экспортировать либо отдельный виджет, либо общий отчёт дашборда (все данные сразу).

## 2. Раздел «Проекты»

- **2.1. Список проектов**
  - В админке отобразить таблицу с колонками: `Название`, `URL (slug)`, `Статус публикации`, `Дата создания`, `Действия`.
  - Добавить сортировку/фильтры по статусу и дате.
- **2.2. Форма редактирования проекта**
  - Разбить на блоки: базовая информация (slug, статус, дата), медиа (обложка, галерея), контент (мультиязычный), отзыв, стек технологий.
  - `slug` — валидировать (латиница/цифры/дефис, уникальность).
- **2.3. Обложка и галерея**
  - Перейти с "вставка ссылки" на загрузку файлов: до 10 МБ каждый и до 20 штук в проекте.
  - На бэкенде: сохранить файлы в `public/uploads/projects/{id}/...` с генерацией относительных URL и базовой валидацией MIME.
  - В БД: хранить в `cover_image` (обложка) и `gallery_json` (массив URL + подписи).
- **2.4. Статус публикации**
  - Заменить выпадающий список на тумблер (switch) `Опубликовано: Да/Нет` с мгновенным сохранением (PATCH‑запрос).
- **2.5. Мультиязычный контент**
  - В `project_locales` использовать существующие поля для языков RU/KA/HY/BG: заголовок, краткое описание (для карточки), полное HTML‑описание.
  - В UI — табы по языкам, с отдельным редактором для каждого.
- **2.6. Редактор текста и карусель внутри описания**
  - Подключить полноценный WYSIWYG (например, TipTap/Quill/TinyMCE) c поддержкой списков, заголовков, ссылок, цитат.
  - Добавить вставку изображений в текст: загрузка файла в тот же каталог `uploads/projects/{id}/content` и вставка `<img>`/специального блока с данными.
  - На фронтенде публичного сайта — реализовать компонент "карусель" для картинок, помеченных особыми классами/структурой.
- **2.7. Отзыв клиента и логотип**
  - В БД создать сущность отзыва, привязанную к проекту: таблица `project_reviews` (если нет или расширить существующую) с полями: рейтинг, текст, язык, ссылка на клиента.
  - В форме проекта: блок "Отзыв клиента" с выбором клиента/компании, загрузкой логотипа, вводом текста отзыва.
  - Логотип сделать кликабельным: ссылка берётся из профиля клиента (`clients`/`client_profiles`).
- **2.8. Стек технологий**
  - В проектах использовать `stack_json`/`stack_*_json` для хранения массива технологий.
  - В админке — справочник технологий (язык, платформа, БД, деплой, мобильные и т.п.) с иконками.
  - В форме проекта — выбор нескольких технологий из справочника по категориям, отображение в публичной части в правой колонке, как на макете.

## 3. Раздел «Статьи»

- **3.1. Список статей**
  - Таблица: `Название`, `URL (slug)`, `Статус публикации`, `Дата`, `Действия`.
- **3.2. Форма статьи**
  - Аналогично проектам: slug, статус (тумблер), дата, обложка, галерея, мультиязычный контент.
  - Загрузка обложек/галереи с лимитом 12 МБ на файл и до ~20 фото.
- **3.3. Редактор статьи**
  - Использовать тот же WYSIWYG, что и для проектов.
  - Поддержка вставки изображений в тело статьи (загрузка, вставка, последующий показ в публичной части).

## 4. Раздел «Калькулятор»

- **4.1. Визуальное обновление**
  - Не трогать текущую бизнес‑логику и API.
  - В админке — только просмотр/фильтрация сессий калькулятора, если нужно.
  - На публичном сайте — привести UI под общую стилистику (отступы, шрифты, цвета), сохраняя все шаги и поля.

## 5. Раздел «Клиенты»

- **5.1. Проверка и доработка учета клиентов**
  - Подтвердить, что создаются записи клиентов для тех, кто прошёл калькулятор/оставил контакты.
  - В списке клиентов добавить фильтры по источнику (калькулятор/форма/ручной ввод).
  - Не менять бизнес‑логику, только UI и, при необходимости, индексы/запросы.

## 6. Раздел «Заказы»

- **6.1. Схема заказов**
  - Убедиться, что таблица `orders` поддерживает связь с `leads`/`clients` и хранит источник заказа.
  - При необходимости добавить поля `source_type` (сайт, знакомые, партнёры и т.п.) и `source_note`.
- **6.2. Создание заказа**
  - В форме заказа реализовать два сценария:
    - выбор клиента из списка существующих (`clients`/`leads`);
    - ввод клиента вручную (имя, контакты, компания) с опцией создать запись в "Клиентах".
  - Выбор источника трафика (сайт, знакомые, реклама и т.д.).
- **6.3. Аналитика по заказам**
  - Добавить отдельную вкладку "Статистика": сводка по источникам, направлениям услуг, статусам заказов.
  - Реализовать графики/таблицы (количество и сумма заказов по времени, по источникам, по направлениям).

## 7. Настройки админ‑панели и безопасность

- **7.1. Проверка всех текущих настроек**
  - Просмотреть существующие разделы настроек, убрать устаревший раздел Email (если он не используется).
- **7.2. 2FA**
  - Доделать и протестировать двухфакторную авторизацию для `admin_users` (на основе уже заложенного TOTP в `admin-routes`).
  - В настройках пользователя — включение/отключение 2FA с показом QR‑кода/секрета и проверкой кода при включении.
  - При входе — выбор режима: только пароль или пароль + 2FA (если включено для пользователя).
- **7.3. UI/UX админ‑панели**
  - Унифицировать шрифты, отступы, цвета, состояния кнопок (primary/secondary/danger, disabled, loading).
  - Вынести повторяющиеся компоненты (таблицы, тумблеры, модальные окна, уведомления) в общий набор.

## 8. Финализация и тестирование

- **8.1. E2E‑проверка сценариев**
  - Протестировать полный цикл: добавление/редактирование проекта, статьи, клиента, заказа, просмотр дашборда, включение 2FA и вход.
- **8.2. Написать краткую документацию**
  - Описать для владельца сайта: как настроить Метрику/Вебмастер в настройках, как добавлять проекты/статьи, как работать с заказами, как включать 2FA.
